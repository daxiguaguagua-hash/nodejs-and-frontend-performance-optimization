## 最小心智模型速查

| 概念 | 心智模型 | 一句话 |
|------|---------|--------|
| 压测 | **餐厅压力测试** | 同时涌进 N 个顾客，看谁先扛不住 |
| wrk 线程/连接 | **厨房和顾客** | `-t` 几个厨房，`-c` 同时多少顾客 |
| CPU 瓶颈 | **厨房出菜慢** | QPS 极低、CPU 高、其他请求全排队 |
| IO 瓶颈 | **等外卖小哥** | QPS 低但 CPU 不高，时间全花在等 |
| PV/UV/QPS | **超市人流量** | PV 进门次数，UV 不同人数，QPS 每秒结账数 |
| 进程/线程 | **厨房和厨师** | 进程=独立厨房，线程=厨房里的厨师 |
| Node 并发 | **唯一服务员** | 不等菜，一桌点完去下一桌，IO 等待中服务他人 |
| libuv 线程池 | **帮厨团队** | Node 自动把 fs/crypto/zlib 丢给 4 个帮厨，主线程不卡 |
| Worker Thread | **临时雇帮厨** | 手动创建，一个 Worker = 1MB 栈，几十个有意义 |
| Go goroutine | **每人自带灶台** | 一个 goroutine = 2KB，10万个不眨眼，runtime 自动分核 |

## Q&A

### Q1: wrk 全称是什么？

wrk 不缩写，就是 "work" 去掉 o。C 语言写的 HTTP 压测工具，系统级工具，不在 `package.json` 里。安装方式：`brew install wrk`。

### Q2: wrk 参数含义

| 参数 | 含义 | 类比 |
|------|------|------|
| `-t` | 线程数 | 几个厨房同时干活 |
| `-c` | 并发连接数 | 同时来了多少顾客 |
| `-d` | 持续时间 | 压力持续多久 |

### Q3: 三种端点压测对比（M4 Mac mini, 16GB）

| 端点 | QPS | 平均延迟 | 瓶颈 |
|------|-----|---------|------|
| `/fast` 纯响应 | 71,862 | 0.15ms | 无（基线） |
| `/cpu-heavy` SHA256×100000 | 28 | 360ms | **CPU**：单线程被计算占满 |
| `/slow-db` setTimeout(50ms) | 190 | 52ms | **IO**：时间花在等，能处理其他请求 |

QPS 跌了 2500 倍！关键公式：`1000ms ÷ 50ms × 10 连接 ≈ 200 QPS`（实测 190，吻合）

### Q4: CPU 瓶颈 vs IO 瓶颈怎么判断？

| 瓶颈类型 | 现象 | 怎么优化 |
|---------|------|---------|
| CPU 瓶颈 | QPS 极低、延迟高、内存平稳 | 优化算法、用 worker thread、换语言 |
| IO 瓶颈 | QPS 低但 CPU 使用率不高 | 加缓存、优化查询、增加连接池 |
| 内存泄漏 | QPS 高但内存持续涨 | 回到 Day 1 排查 |

### Q5: PV / UV / QPS 换算公式

- **PV** = Page View（页面浏览量），刷新一次算一次
- **UV** = Unique Visitor（独立访客），一个人一天只算一次
- **QPS** = Queries Per Second（每秒查询数）

**二八原则**：80% 的流量集中在 20% 的时间段

```
QPS ≈ (日 PV × 0.8) ÷ (86400 × 0.2)
```

速算法：`QPS × 21600 ≈ 日 PV`

| 日 PV | 峰值 QPS |
|--------|---------|
| 100 万 | ~46 |
| 1000 万 | ~463 |
| 1 亿 | ~4,630 |

面试标准回答："按二八原则，80% 流量集中在高峰 20% 时段，日 PV × 0.8 ÷ (86400 × 0.2) = 峰值 QPS，翻一倍做冗余。"

### Q6: Node.js 是单线程的，怎么会有"并发"？

Node.js 的并发 = 事件循环并发，不是多线程并发。IO 密集型场景下，单线程在等待 IO 时去服务其他连接，看起来像同时处理，实际上线程永远是"一件事干完立刻干下一件"。只有 CPU 密集任务才会卡死。

### Q7: 进程 vs 线程

| | 进程 | 线程 |
|---|---|---|
| 类比 | 独立的厨房（全套设备） | 厨房里的厨师（只干活） |
| 内存 | 隔离，不共享 | 同一进程的线程共享内存 |
| 创建开销 | 大 | 小 |
| 崩了 | 不影响其他进程 | 整个进程挂 |

一个 Node 进程 = 一个主线程 + Worker Threads（可选）。Docker 里跑 `node server.js` = 一个进程 = 一个 Node 实例。

### Q8: CPU 密集任务会自动交给 Worker Thread 吗？

**不会。** 必须手动选择异步 API 或手动开 Worker。

| 自动交给 libuv 线程池（4 个线程） | 必须手动处理 |
|----------------------------------|------------|
| `crypto.pbkdf2()` 密码哈希 | `crypto.createHash().digest()` 同步哈希 |
| `fs.readFile()` | `for` 循环大量计算 |
| `zlib.gzip()` | `sharp` 图片处理 |
| `dns.lookup()` | 超大 JSON 序列化 |

### Q9: 异步 vs 同步哈希压测对比

| 端点 | 方式 | QPS |
|------|------|-----|
| `/cpu-heavy` | `createHash()` 同步 | 28 |
| `/cpu-heavy-async` (默认 4 线程) | `pbkdf2()` 异步 | 208 |
| `/cpu-heavy-async` (8 线程) | `pbkdf2()` + `UV_THREADPOOL_SIZE=8` | 273 |

异步版本 7.4 倍提升！但 273 远远低于 `/fast` 的 71862，因为线程池只有 8 个线程，第 9 个请求开始排队。

### Q10: Worker Thread 手动池实战对比（Day 2 完整总结表）

4 种方式处理同量级 SHA256 计算（10万次），8 线程池：

| 端点 | 方案 | QPS | 倍数 | 原理 |
|------|------|-----|------|------|
| `/fast` | 纯响应（基线） | 71,862 | — | 无计算 |
| `/cpu-heavy` | 同步主线程 | 28 | 1× | 主线程被算死 |
| `/cpu-heavy-worker` | Worker 池 | **122** | 4.3× | 8 Worker 分担，但仍是 JS 跑 |
| `/cpu-heavy-async` | pbkdf2 (4线程) | 208 | 7.4× | C 级 OpenSSL 加速 |
| `/cpu-heavy-async` | pbkdf2 (8线程) | 273 | 9.8× | 线程翻倍 + 原生 C |

**核心洞察**：Worker 解决的是"不卡主线程"，不是"加速计算"。`pbkdf2` 更快是因为它调了 OpenSSL 的 C 代码而非 V8 跑 JS。

### Q11: Worker 为什么只做到 122 QPS，比 pbkdf2 的 273 差？

因为 `heavy-worker.js` 里还是用 JS 跑 `createHash()`：

- Worker：JS 循环 SHA256 → V8 解释执行 → 只是换了个线程，计算本身没加速
- pbkdf2：Node → OpenSSL C 库 → 真·底层加速
- `postMessage()` 还有序列化开销

所以正确姿势优先级：**原生异步 API > Worker 池 > 换语言**

### Q12: Node vs Go 的分水岭在哪里？

| | Node Worker Thread | Go goroutine |
|---|---|---|
| 单个开销 | ~1MB 栈 | **~2KB** 栈 |
| 能同时开 | 几十个有意义 | 10 万个不眨眼 |
| 通信 | `postMessage()` 序列化拷贝 | channel 零拷贝 |
| 全核调度 | 手动管理池 | runtime 自动分核 |

三条分水岭：
1. **并发数量**：BFF 100 并发 → Node 够。计算密集型 10000 并发 → Go 闭眼跑，Node 崩。
2. **通信密度**：频繁在多个 Worker/goroutine 之间传数据 → Node 死在序列化上。
3. **全核并行**：16 核服务器跑大数据 → Go 自动分核，Node 手动管池。

**结论**：不是 Go 更快，是它们优势区不同。Node 优势 = IO 密集/少计算/多聚合。Go 优势 = 多计算/高并发/密集通信。

### Q13: 面试时怎么跟 Go/Java 面试官谈 Node？

不要谈语言好坏，谈职责匹配：

| 对方问 | 你答 |
|--------|------|
| "Node 单线程能扛并发？" | "BFF 层 80% 请求是 IO 聚合，事件循环 QPS 远超需要，瓶颈在下游服务不在 BFF。" |
| "Node 做计算不行吧？" | "CPU 密集不在 Node 做。Go 服务算完，Node 拼好给前端。各干各的。" |
| "Node 生态乱？" | "BFF 胶水层轮子 Node 最全。数据聚合、鉴权、缓存。" |

知道自己的技术边界比吹"Node 无敌"值钱。Go/Java 大佬尊重的是"我知道工具该用在哪"。

### Q14: 并发连接数 ≠ QPS

并发连接 = 同一时刻在处理的请求数。QPS = 每秒能处理完多少个。

```
QPS = 并发连接数 ÷ 平均响应时间
如果 BFF 一个请求 10ms → 100 并发 → QPS = 10000
如果 BFF 一个请求 50ms → 100 并发 → QPS = 2000
```

并发连接看用户侧同时打开的请求。消息队列是异步的，写操作走队列反而不占并发。

### Q15: wrk 输出报告的每一行怎么读？（体检报告模型）

wrk 的数据像体检报告 —— 每一项告诉你哪个零件有问题。

```
Thread Stats   Avg      Stdev     Max   +/- Stdev
  Latency    81.11ms   44.67ms 173.80ms   57.00%
  Req/Sec    61.80     19.22   111.00     72.22%
```

**Latency（延迟）**：

| 指标 | 大白话 | 健康标准 |
|------|--------|---------|
| Avg | 平均每个请求等了多久 | 越小越好，> 1s 用户体验差 |
| Stdev | 波动幅度 | 越小越稳定 |
| Max | 最惨的那个等了多久 | 与 Avg 差 2~3 倍以内正常 |
| +/- Stdev | 多少比例在 Avg±Stdev 范围内 | > 90% 最好，< 70% 性能不均 |

**Req/Sec（吞吐）**：

| 指标 | 大白话 |
|------|--------|
| Avg | 每秒平均处理了多少请求 |
| Stdev | 每秒处理数波动，大说明不稳定 |
| Max | 某一秒爆发处理了多少 |
| +/- Stdev | 稳定时间段的比例 |

**QPS 计算**：`Requests/sec = 总请求数 ÷ 总时间`。例：`122.77 = 1238 ÷ 10.08s`

### Q16: Stdev / P50 / P99 怎么理解？

最小心智模型：**P 就是"排队"。P50 = 如果你拉了 100 个人排队，第 50 个人等了多久。P99 = 第 99 个人等了多久。**

wrk 本身不输出 P50/P99，需要用其他工具（wrk2、autocannon）或对 wrk 结果做估算：

- **P50（中位数）** ≈ Avg，如果 +/- Stdev 高说明接近正态分布
- **P99** ≈ Max × 0.7~0.9，用于判断"最差的那 1% 用户体验"
- **如果 Stdev > Avg** → 性能分布极度不均 → 很可能有偶发阻塞

面试一句话："P50 看大多数用户体验，P99 看长尾问题。P99 过高通常说明有间歇性阻塞或 GC 暂停。"

```
├─────────┼──────┼──────┼───────┼──────┼─────────┼─────────┼──────┤
│ Latency │ 0 ms │ 0 ms │ 0 ms  │ 0 ms │ 0.01 ms │ 0.02 ms │ 4 ms │
└─────────┴──────┴──────┴───────┴──────┴─────────┴─────────┴──────┘
  QPS: 67,890 | Req/Sec Stdev: 1,405 (对 68K QPS 来说极小)
```

**三种端点完整对比：**

| | `/fast` | `/cpu-heavy-worker` | `/cpu-heavy` |
|---|---|---|---|
| P50 | 0ms | 83ms | — |
| P99 | 0ms | 161ms | — |
| Max | 4ms | 174ms | 1.76s |
| QPS | 67,890 | 100 | 28 |
| P50→P99 差距 | 0 倍 | **2 倍** | 巨大 |
| 瓶颈 | 无 | Worker 排队 | 主线程卡死 |

**核心规律**：
- 好的端点：P50 和 P99 差距 1~2 倍。差距越大排队越严重。
- P50 看大多数体验，P99 看有没有人在骂你。
- Avg 会骗人——P99 160ms 的用户已经不爽了，Avg 78ms 根本看不出来。
- 服务刚重启时会出现"脉冲"：Worker 同时完成、憋着的请求一起释放，Req/Sec Stdev 飙到 204。等稳态后就稳了（Stdev 7.61）。

### Q18: `-t/-c` 不同组合对 QPS 的影响规律（完整实验）

测 4 组，同一 `/fast` 端点（M4 Mac mini, Express）：

| 组合 | QPS | Avg | Stdev | Max | 判断 |
|------|-----|-----|-------|-----|------|
| `-t2 -c10` | 56,065 | 0.18ms | 0.07ms | 4.4ms | 正常 |
| `-t2 -c50` | 57,632 | 0.88ms | 0.44ms | 27.2ms | 排长队 |
| `-t4 -c100` | 57,979 | 1.85ms | **2.38ms** | **84.9ms** | 临界 |
| **规律** | **QPS 不涨** | **延迟 10×** | **Stdev > Avg** | **最差差 20×** | — |

QPS 三组都一样（~57K），连接从 10 涨到 100 并没带来更多吞吐，只让最惨的用户从 4ms 等到 85ms。

**规律**：

```
-c < 临界点：  -c ↑ → QPS ↑，延迟微涨
-c ≈ 临界点：  -c ↑ → QPS 不涨，延迟猛涨（全在排队）
-c > 临界点：  -c ↑ → QPS 开始跌，错误出现（系统压垮）
```

**Stdev > Avg** 是进入崩溃区的边缘信号 —— 多数人还好，少数人极度恶化。

**`-t` 的作用**：只控制 wrk 自己的发送线程，不影响服务器。`-t` ≥ CPU 核数就能榨干 wrk 的发送能力。

**压测标准组合策略**：
1. `-c10` 定基线
2. 翻倍 `-c` 直到 QPS 不涨 → 找到极限 QPS
3. 继续翻倍直到错误率 > 1% → 找到崩溃点
4. 极限 QPS × 70% 设定生产告警线

注：wrk 的输出格式是机器可读风格，非表格。这是刻意的——wrk 设计为数据管道工具，预期输出会被转到 HTML 监控面板或其他可视化系统呈现。

### Q19: JMeter 是什么？

Apache 出的 Java 写的企业级压测工具，对比 wrk：

| | wrk | JMeter |
|---|---|---|
| 语言 | C（145KB） | Java（几百 MB，需 JDK） |
| 界面 | 命令行 | 图形界面 |
| 场景 | 单一 URL 猛砸 | 多接口混合、递增压力、模拟登录→下单→支付 |
| 报告 | 纯文本/JSON | HTML 图表（饼图、柱状图、趋势图、APDEX评分） |
| 使用场景 | 开发自测、快速摸底 | 给领导看的压测报告 |
| 学习成本 | 5 分钟 | 半天 |

BFF 开发自测用 wrk + autocannon 足够。JMeter 的核心价值 = 生成漂亮 HTML 报告给领导看 + 复杂业务流程压测。

### Q20: 错误率分析 — 压测时出 4xx/5xx 说明什么？

| 状态码 | 含义 | 压测中怎么判断 |
|--------|------|-------------|
| 2xx | 正常 | 理想状态 100% |
| 4xx | 客户端错误 | 请求有问题（URL错、参数错） |
| 5xx | 服务端错误 | **系统被打爆了** — 连接池耗尽、OOM、超时 |
| Non-2xx | wrk 汇总 | 所有非成功请求的总数 |

之前 `pbkdf2` 实验 `Non-2xx: 629259` 是因为参数类型导致回调报错（服务端 5xx）。正确判断：
- 错误率突然从 0% 开始涨 → 正在进入崩溃区
- 错误率 > 1% → 已经过了极限，立即降压
- 100% 错误 → 服务已经挂了，不是压测问题

### Q15 续: Node.js 的正确定位

Node.js 的职责不在后端核心计算，它应该做 **BFF（Backend For Frontend）**：数据聚合、鉴权、请求校验等无状态任务。CPU 密集型不该交给 Node。
