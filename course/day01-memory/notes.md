# Day 1 笔记：内存管理与内存泄漏

## 最小心智模型速查

| 概念 | 心智模型 | 一句话 |
|------|---------|--------|
| V8 分代 GC | **草稿纸 + 档案柜** | 新生代=草稿纸(快清)、老生代=档案柜(满了才清) |
| 内存泄漏 | **引用忘了断** | GC 不猜你还要不要，只看引用链断了没 |
| 全局数组泄漏 | **永不关门的仓库** | 全局变量活着，往里塞的东西 GC 都不敢扔 |
| 闭包泄漏 | **钥匙和上锁的房间** | 内层函数是钥匙，它活着房间里的东西全锁着 |
| 进程数 > 核数 | **厨师抢灶台** | 厨师(进程)多过灶台(CPU核)就互相撞，卡死 |
| Node.js 事件循环 | **高速公路单车道** | 主线程一次只做一件事，底层就是流式处理 |
| 堆外内存 | **仓库外面的院子** | Buffer 不归 V8 GC 管，泄漏会直接撑爆物理内存 |

## Q&A

### Q1: `process.memoryUsage()` 的 5 个字段分别是什么意思？

| 字段 | 含义 | 大白话 |
|------|------|--------|
| `rss` | Resident Set Size，常驻内存集 | 操作系统给这个进程分配了多少物理内存 |
| `heapTotal` | 堆总量 | V8 引擎从系统申请了多少堆空间 |
| `heapUsed` | 堆已用 | 实际用掉了多少堆空间 |
| `external` | 外部内存 | C++ 对象占的内存（不在 V8 堆里） |
| `arrayBuffers` | 数组缓冲 | ArrayBuffer 占的内存 |

**正常曲线**：heapUsed 涨涨涨跌涨涨涨跌 —— 锯齿状，说明 GC 在工作。
**泄漏信号**：heapUsed 只涨不跌 —— 内存回收不了。

### Q2: 纯数字看不到锯齿怎么办？

用 ASCII 字符画走势图（Unicode block characters: ▁▂▃▄▅▆▇█），配合 `console.clear()` 刷新终端，模拟实时图表。

也可以用 `node --inspect` + Chrome DevTools Memory 面板看真实的图形化内存曲线。

### Q3: 为什么内存监控里 heapUsed 一直在涨，没看到 GC 回收？

V8 的 GC **不是定时执行的**，而是"堆快满了才出手"。

默认老生代堆上限约 1.4GB，如果堆只用了几 MB，V8 觉得没必要急着回收，所以看起来一直涨。

**解决办法**：降低堆上限，强制 V8 更频繁 GC：
```
node --max-old-space-size=16 day01-memory/01-basic-server.js
```
把老生代堆限制到 16MB，堆小了 GC 就会频繁触发，锯齿就出来了。

### Q5: 云函数会用 `--max-old-space-size` 限制内存吗？

不会直接用它，但原理一样。云函数通过平台配置限制**总内存**（如 256MB），V8 堆自动 = 总内存 × 50%~70%。内存越小 GC 越勤快，泄漏越快暴露。

### Q6: 新生代和老生代 GC 是什么关系？（最小心智模型）

**草稿纸 + 档案柜**：
- 新生代 = 考场的草稿纸（新对象，几 MB，满了就 Scavenge 快速回收）
- 老生代 = 家里的档案柜（老对象，满了才 Mark-Sweep 大扫除）
- 对象旅程：`new → 草稿纸 → 活过两次 GC → 晋升到档案柜`
- `--max-old-space-size=16` 限制的是"档案柜"的大小

### Q7: 内存泄漏的本质是什么？

**引用忘了断**。GC 不是 AI，它只判断"有没有引用链"：
- 有引用 → 留着
- 没引用 → 回收
泄漏 = 你不再需要它了，但引用链还在，GC 以为它有用。
三种典型：全局变量不释放、闭包握住大对象、定时器没 clear。

### Q8: Node.js 核数从 12 调到 6 就不卡了是怎么回事？

**厨房模型**：厨师数 > 灶台数 → 互相撞。进程数超过物理核数导致 CPU 疯狂切换（上下文切换开销） + 内存叠加爆满。规则：进程数 ≤ 物理核数。

### Q9: 带 `--max-old-space-size=16` 时内存也会回落，说明泄漏被修好了吗？

没有。回落的是**临时垃圾**（请求对象、中间变量），泄漏的数据一个没少。关键指标是**数组长度只涨不降** + **低谷线在不断抬高**。不带限制的话低谷抬升太慢看不出来，但最终也会 OOM。

### Q10: 全局数组泄漏 8 次 batch/100 的实战数据说明了什么？

| 批次 | 堆已用 | 数组长度 |
|------|--------|---------|
| 1    | 8.92   | 100     |
| 2    | 10.11  | 200     |
| 3    | **9.64** ↓ | 300  ← GC 清了临时垃圾，但数组一个没少 |
| 4    | 10.72  | 400     |
| 5    | 11.74  | 500     |
| 6    | 12.72  | 600     |
| 7    | 12.81  | 700     |
| 8    | 13.80  | 800     |

堆有涨有跌（GC 在工作），数组长度只涨不跌（泄漏的数据 GC 不收），低谷线持续抬高，最终会突破 16MB 上限 crash。

### Q11: Node.js 是面向对象/函数式的，为什么说它内里是"流式处理"？

**高速公路单车道**：Node.js 主线程 = 事件循环 = 一条无尽的单车道。每次从队列取一个任务，执行完，取下一个。底层就是"一件事一件事地过"，跟 C 的流式处理本质一样。闭包泄漏在单线程里尤其严重——一条流水线堵了，后面的全排队。

### Q12: 03 闭包泄漏实验中，`/subtle-leak` 的 50MB Buffer 为什么不触发 OOM？

`Buffer.alloc(50MB)` 走的是**堆外内存**（`arrayBuffers`），不归 `--max-old-space-size=16` 管。`--max-old-space-size` 只限制 V8 堆（JS 对象、数组、闭包），拦不住 Buffer。生产上 Buffer 泄漏会直接撑爆物理内存，比 JS 对象泄漏更危险。

```
V8 堆内存（受 --max-old-space-size 限制）
├── JS 对象、数组、闭包  ← OOM 在这里触发
└── ...

堆外内存（不受限制）
└── Buffer、ArrayBuffer  ← 不看堆大小，看物理内存
```

### Q13: 03 闭包泄漏实战，`/leak` 端点刷了几次后 heapUsed 从 7.79 → 8.25 MB

每次访问 `/leak` 都执行 `createHandler()`，在闭包环境里放入 `new Array(100000)`，然后只返回一个元素。闭包（内层函数）作为 Express 路由处理器一直活着 → 那 10 万个元素的数组一个都扔不掉。每次请求涨 ~0.5MB，持续刷会一直涨到 OOM。

### Q14: 04 定时器泄漏实验总结

主 `setInterval` 每秒写入 5KB 到全局数组，启动即开始漏。`/leak` 端点每次请求创建一个不会 stop 的 interval，进一步加速泄漏。`/stop-main-leak` 可以停掉主泄漏，但 `/leak` 创建的那些没有引用无法停止，会一直跑。

## Day 1 总结：前端视角对照表

| Day 1 学会的概念 | 在你的前端工作中对应什么 |
|-----------------|------------------------|
| GC 锯齿曲线 | 用 Chrome Performance 面板看内存时能判断是否有泄漏 |
| 全局变量泄漏 | React 里模块级 `const cache = new Map()` 不清理同理 |
| 闭包泄漏 | `useEffect` 里没清定时器、`useCallback` 引用大对象 |
| 定时器泄漏 | `setInterval` 没 `clearInterval`、事件监听没 `removeEventListener` |
| 堆外内存 | 前端少碰，但做 Node BFF/SSR/中间层时要注意 Buffer 和 Stream |
